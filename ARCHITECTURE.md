# Архитектура Проекта

Документ описывает текущее состояние архитектуры и реальные связи между модулями.

## 1. Архитектурный стиль

Проект реализован как модульный монолит на чистом JavaScript без сборки.

- Один рантайм в браузере.
- Четкое разделение ответственности между файлами.
- Публичные точки входа для HTML через глобальные callbacks.

## 2. Слои и зависимости

```text
index.html
  -> script.js (callbacks, orchestration)
      -> UIManager (ui-manager.js)
      -> GameEngine (game-engine.js)
          -> RoleUtils/RoleHandlers (roles.js)
          -> ConfigUtils/GAME_CONFIG (config.js)
```

Направление зависимостей строго сверху вниз. UI не хранит доменную логику ролей, а движок не рендерит интерфейс.

## 3. Ответственность модулей

### config.js

- GAME_CONFIG:
  - ROLES;
  - NIGHT_ROLE_ORDER;
  - NIGHT_ROLE_BRIEFS;
  - DEFAULT_ROLE_CONFIG;
  - WIN_CONDITIONS;
  - PHRASES.
- ConfigUtils для безопасного доступа к конфигу и фразам.

### roles.js

- RoleHandlers по одной стратегии на роль.
- RoleUtils как фасад для движка.

Типовой контракт роли:

- canSelectTarget(targetIdx, gameState)
- onNightAction(targetIdx, gameState)
- getRestrictions()

### game-engine.js

Центр бизнес-логики:

- жизненный цикл партии;
- валидация конфигурации ролей;
- генерация и применение рекомендаций по ролям;
- ночные действия и резолв атак/защит;
- дневное голосование и ничьи;
- проверка победы;
- логирование и экспортный формат.

### ui-manager.js

Центр рендера и пользовательских экранов:

- управление экранами s0..s6;
- карточка рекомендации ролей;
- ручной конструктор ролей;
- шапка и краткое саммари по живым ролям;
- модалки правил, лога и сообщений;
- экспорт текстового лога.

### script.js

Интеграционный слой:

- инициализация приложения;
- экспорт функций в window для inline-onclick;
- orchestration между UIManager и GameEngine;
- защита от частых повторных тапов;
- переходы между фазами и экранами.

## 4. Поток данных по фазам

### Подготовка

1. Пользователь меняет состав игроков.
2. На экране ролей UI вызывает GameEngine.getRoleRecommendations.
3. Рекомендация автоматически применяется при первом входе на шаг ролей для текущего числа игроков.
4. Пользователь может вручную скорректировать роли.

### Раздача

1. Проверяется валидность конфигурации.
2. Формируется порядок раздачи ролей.
3. Роли назначаются игрокам на отдельном экране.

### Ночь

1. GameEngine.startNight формирует активные роли среди живых.
2. Роли ходят по NIGHT_ROLE_ORDER.
3. Действия записываются в nightActions.
4. GameEngine.processNight рассчитывает итог ночи.

### День

1. Голосование с ограничением по живым игрокам.
2. При первой ничьей запускается переголосование между лидерами.
3. При второй ничьей никто не изгоняется.

### Финал

Победа проверяется после ключевых событий:

- после ночи;
- после дневного изгнания.

## 5. Инварианты домена

- Спецроли не могут занять весь стол: должен остаться минимум один Мирный.
- В партии всегда минимум одна Мафия.
- Босс Мафии не может быть больше одного.
- Любовница невозможна без Детектива.
- Доктор не лечит одну цель две ночи подряд.
- Детектив не проверяет себя и повторно ту же цель.
- Телохранитель не защищает себя.
- Счастливчик один раз переживает смертельную ночную атаку.

## 6. Рекомендательная система ролей

В движке используется перебор валидных конфигураций с расчетом score.

Учитываются:

- баланс сил города и маф-стороны;
- доля мирных;
- насыщенность ночными ролями;
- разнообразие ролей;
- бонусы за живой геймплейные связки.

Результат:

- recommended: лучшая конфигурация;
- reason: объяснение выбора для UI.

## 7. Логирование

Лог хранится в движке как структурированные записи:

- timestamp;
- text;
- phase;
- night;
- scope.

UI отображает timeline-лог с группировкой по фазе и номеру ночи. Экспорт выполняется в plain text.

## 8. Тестируемость

autotest.js запускает движок в node-контексте и проверяет ключевые доменные сценарии без браузера.

Это дает:

- быструю регрессионную проверку правил;
- изоляцию бизнес-логики от UI;
- безопасное внесение изменений в роли и победные условия.

## 9. Расширение системы

Чтобы добавить новую роль без ломки архитектуры:

1. Описать роль и фразы в config.js.
2. Добавить RoleHandler в roles.js.
3. Добавить роль в NIGHT_ROLE_ORDER, если она ночная.
4. При необходимости поправить экранные ограничения в ui-manager.js.
5. Добавить тесты в autotest.js.
